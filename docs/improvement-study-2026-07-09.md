# 개선 방안 연구 (2026-07-09)

BYOK 전환·Vercel 배포 직후 수행한 코드베이스 분석. 아래 "확인됨"으로 표시한 항목은 실행 가능한 프로브 테스트나 직접 코드 확인으로 검증했다. 시장·저작권·Gemini 무료 티어 한도에 대한 외부 조사는 **수행하지 못했다**(세션 한도로 병렬 리서치가 중단됨). 따라서 이 문서는 코드에 근거한 결론만 담는다.

> **정정 (2026-07-09).** 이 문서의 초판은 "옛한글이 .pptx에서 두부(tofu)로 깨진다"고 적었다. **틀렸다.** 소유자가 실물 산출물에서 정상 렌더링을 확인했다. 폰트가 아니라 **폭 계산**이 문제였다 — 정확한 내용은 1-4에 다시 썼다. 렌더링과 메트릭스를 혼동한 오류였다.

## 1. 지금 고쳐야 할 것

### 1-1. 같은 단어가 여러 번 나오면 주석이 첫 번째 것에만 붙는다 — 확인됨

`src/lib/annotation-matcher.ts:25`

```ts
const exactIdx = fullText.indexOf(targetText);
```

`indexOf`는 첫 번째 출현만 찾는다. Gemini가 `targetText: "구보"`인 주석을 두 개 반환하면 **둘 다 같은 자리(0~2)로 매칭된다.**

프로브 실행 결과:

```
입력: "구보는 다시 걷기로 한다.\n밤이 되어 구보는 집으로 돌아왔다."
      주석1 = 구보 → "첫 등장 — 산책의 시작" (circle)
      주석2 = 구보 → "귀가 — 하루의 끝"     (rectangle)

출력: matched spans: [ [0,2,'첫 등장…'], [0,2,'귀가…'] ]   ← 둘 다 [0,2]
      unmatched: 0                                        ← 실패로 잡히지도 않음
```

두 번째 "구보"(인덱스 17)에는 아무 표시도 붙지 않는다. 게다가 `src/lib/html-generator.ts:42`의

```ts
if (start >= end) continue;
```

때문에 겹친 두 번째 주석은 렌더 단계에서 **조용히 사라진다.** 오류도, 경고도 없다.

교사에게 벌어지는 일: 소설에서 반복되는 인물명(구보, 그)이나 시에서 반복되는 시어에 붙은 교사용 교과서 주석이 엉뚱한 위치에 찍히거나 통째로 없어진다. 실제로 이 도구의 E2E 검증에서 생성한 「소설가 구보씨의 일일」 슬라이드가 정확히 이 패턴이었다.

고칠 방향: 매칭을 **출현 순서 인식**으로 바꾼다. 최소 수정은 `targetText`별 커서를 유지해 N번째 동일 주석이 N번째 출현을 가져가게 하는 것(주석은 대체로 읽기 순서로 나온다). 더 견고한 방법은 Gemini 응답 스키마에 `occurrenceIndex` 또는 `precedingContext`(앞 10자)를 추가해 위치를 특정하는 것이다. 후자가 정답에 가깝고, 프롬프트 한 줄과 스키마 필드 하나면 된다.

난이도: S~M.

### 1-2. 슬라이드 경계를 걸치는 주석은 뒷부분이 사라진다 — 확인됨

`src/lib/annotation-matcher.ts:148, 156`

```ts
belongs = ann.startIndex >= slideStart && ann.startIndex < slideEnd;   // 시작 위치로만 판정
endIndex: Math.min(ann.endIndex, slideEnd) - slideStart,               // 뒤는 잘라냄
```

시작점이 속한 슬라이드에만 배정하고 끝은 잘라낸다. 잘려나간 뒷부분은 다음 슬라이드로 넘어가지 않는다.

프로브 실행 결과:

```
슬라이드1 주석: [ [3, 7, '줄입니다'] ]   ← 앞부분만
슬라이드2 주석: []                      ← 뒷부분 소실
```

교사에게 벌어지는 일: 연이나 문단 경계를 걸치는 밑줄·꺾쇠가 반쪽만 그려진다. 산문에서 글자 수 기준으로 자동 분할하므로 경계는 문장 한가운데 떨어질 수 있다.

고칠 방향: 걸치는 주석을 두 개로 쪼개 양쪽 슬라이드에 각각 배정하되, 마커 도형만 이어 그리고 주석 텍스트는 시작 슬라이드에 한 번만 붙인다. 혹은 분할 지점을 주석 경계로 스냅시킨다(더 단순하고 교육적으로도 자연스럽다).

난이도: M.

### 1-3. 저장·불러오기가 PPT 설정을 통째로 날린다 — 확인됨

`pptSettings`가 없다. 그래서 `exportProject`(:457)와 `importProject`(:462), 그리고 2초 디바운스 자동저장·복원 경로 모두 **글꼴 크기·줄간격·슬라이드 크기·주석 글꼴 크기를 기본값으로 되돌린다.**

### 1-4. 옛한글은 잘 그려지지만, 폭을 절반으로 계산한다 — 확인됨

`src/lib/pptx-geometry.ts`의 `getCharWidthInch`:

```ts
if (/[가-힯一-鿿㐀-䶿＀-￯]/.test(char)) return ptToInch * m.hangul;
if (/[ㄱ-ㆎ　-〿]/.test(char)) return ptToInch * m.hangul;
...
return ptToInch * m.latinLower;   // ← 여기로 떨어진다
```

첫 정규식은 **완성형 한글 음절**(U+AC00–D7AF), 둘째는 **호환 자모**(U+3131–318E)를 잡는다. 그런데 아래아가 든 옛한글 음절은 유니코드에 완성형이 없어 **조합용 자모**(U+1100–U+11FF)로 인코딩된다. 두 정규식 모두 이 블록을 포함하지 않는다. 따라서 옛한글 자모는 마지막 줄의 `latinLower`(0.5em)로 떨어진다.

한 음절이 자모 2~3개이므로 계산된 폭은 **1.0~1.5em**, 실제는 **0.932em**이다. 브라우저와 PowerPoint는 폰트가 시키는 대로 정상 렌더링하지만, 앱은 그 글자가 얼마나 넓은지 틀리게 안다. 결과: 옛한글이 든 행에서 줄바꿈 지점이 어긋나고, 밑줄 도형의 길이와 마커의 X 좌표가 실제 글자와 맞지 않는다.

고칠 방향: 한글 정규식에 `ᄀ-ᇿ`(조합 자모)와 `ꥠ-꥿`·`ힰ-퟿`(확장 A/B)를 추가한다. 다만 조합 자모는 여러 개가 모여 한 음절이 되므로 개당 `hangul` 폭을 주면 이번엔 과대 계산된다. 정확히 하려면 `Intl.Segmenter`(grapheme)로 음절 단위를 센 뒤 음절당 `hangul` 폭을 적용해야 한다. 난이도 M.

### 1-5. 슬라이드를 병합하면 클릭 순서가 중복된다 — 확인됨

`src/lib/slide-splitter.ts:262-269`

```ts
annotations: [
  ...first.annotations,
  ...second.annotations.map((a) => ({ ...a, startIndex: a.startIndex + firstLen, endIndex: a.endIndex + firstLen })),
],
```

`startIndex`/`endIndex`는 밀어주는데 `order`는 그대로다. 첫 슬라이드가 1,2이고 둘째가 1,2,3이면 병합 결과는 **1,2,1,2,3**이다. 클릭 애니메이션은 `order`로 순서를 정하므로, 교사가 슬라이드를 합치는 순간 등장 순서가 뒤엉킨다.

고칠 방향: 병합 후 `merged.annotations.forEach((a, i) => a.order = i + 1)`. 난이도 S.

## 2. 폰트를 바꿀 수 있는가

**결론부터: 구조는 이미 준비돼 있지만, 지금 바꾸면 어긋난다.** 지뢰가 두 개 있고 둘 다 작다.

### 2-1. 이미 되어 있는 것

`src/lib/font-metrics.ts`가 폰트별 메트릭스 테이블을 갖고 있다 — `한컴산뜻돋움`, `맑은 고딕`, `나눔고딕` 세 종. `getFontMetrics(settings.fontFamily)`로 조회한다. 즉 폰트는 **이미 파라미터**다. 하드코딩이 아니다.

실제 폰트는 `src/lib/types.ts:77,87`의 `fontFamily: "한컴산뜻돋움"`에서 온다. UI에는 폰트 선택기가 없다.

### 2-2. 지뢰 하나: 죽은 상수

`src/lib/pptx-constants.ts:31`의 `export const FONT_FAMILY = "한컴산뜻돋움"`은 **어디서도 import되지 않는다.** 이걸 고쳐도 아무 일도 일어나지 않는다. 같은 파일 `:48`의 `FONT_LINE_HEIGHT_RATIO = 1.3`(한컴산뜻돋움 전용 값)도 주석 외에는 쓰이지 않는다. 지워야 한다.

### 2-3. 지뢰 둘: 같은 값을 두 파일이 다르게 계산한다

```
src/lib/pptx-geometry.ts:127, :194   →  fontSize * PPT_LINE_STEP_RATIO * lineSpacing   (1.22 하드코딩)
src/lib/pptx-generator.ts:275        →  fontSize * metrics.lineStepRatio * lineSpacing (테이블 조회)
```

한컴산뜻돋움은 테이블 값도 1.22라 **두 경로가 우연히 일치한다.** 그래서 지금은 아무 증상이 없다. 폰트를 바꾸는 순간 갈라진다.

32pt 기준 실측 계산:

| 폰트 | lineStepRatio | 줄당 어긋남 | 7줄 누적 |
|------|---------------|------------|---------|
| 한컴산뜻돋움 | 1.22 | 0.00mm | 0.00mm |
| 나눔고딕 | 1.24 | 0.23mm | 1.58mm |
| 맑은 고딕 | 1.25 | 0.34mm | 2.37mm |

본문 텍스트는 PowerPoint가 그리고 마커 도형은 우리가 좌표로 찍으므로, 이 차이만큼 **밑줄이 글자에서 아래로 밀린다.** 7줄짜리 슬라이드 마지막 줄에서 2.4mm면 교실 스크린에서 명확히 보인다.

고칠 방향: `pptx-geometry.ts`의 두 곳에 `metrics`를 넘겨 `metrics.lineStepRatio`를 쓰게 한다. 두 줄이다. 그러면 폰트 교체가 안전해진다 — **단, 테이블의 메트릭스 값이 맞다는 전제에서.**

### 2-4. 새 폰트를 추가하려면

`FONT_METRICS`에 항목 하나를 더하면 된다. 값은 추측하지 말고 폰트 파일에서 읽어야 한다. `opentype.js`나 `fontkit`으로:

- `hangul` = `'가'`의 advance width ÷ `unitsPerEm`
- `latinUpper`/`latinLower`/`digit`/`space`/`punctuation`/`bracket` = 각 대표 글자의 advance ÷ `unitsPerEm`
- `lineStepRatio` = 실측 보정값. PowerPoint의 실제 줄간격은 `(usWinAscent + usWinDescent) / unitsPerEm`보다 약간 작다. 슬라이드 하나에 4줄 이상을 찍어 마지막 줄의 드리프트가 0이 되도록 맞춘다.

### 2-5. 그런데 정말 바꿔야 하는가

`한컴산뜻돋움`은 한컴오피스와 함께 설치되는 폰트다. **한컴오피스가 없는 컴퓨터에서 `.pptx`를 열면 PowerPoint가 다른 폰트로 대체하고, 그 순간 우리가 계산한 모든 마커 좌표가 틀어진다.** pptxgenjs는 폰트를 임베딩하지 못한다. 지금 소유자의 컴퓨터에서 결과물이 정확한 이유는 그 폰트가 설치돼 있기 때문이다.

이건 배포 대상이 넓어질수록 커지는 문제다. 선택지는 셋이다.

첫째, `맑은 고딕`으로 바꾼다. Windows Vista 이후 모든 Windows에 있으므로 대체가 일어나지 않는다. 대신 **아래아를 담고 있지 않다** — 옛한글 작품에서는 못 쓴다.

둘째, 폰트를 설정으로 노출하고 기본값만 `맑은 고딕`으로 둔다. 옛한글이 필요한 교사만 한컴 폰트를 고른다. 2-3을 고친 뒤에야 안전하다.

셋째, 지금처럼 둔다. 대신 README와 생성 완료 화면에 "이 파일은 한컴산뜻돋움이 설치된 컴퓨터에서 열어야 배치가 정확합니다"라고 **명시한다.** 코드 변경 0줄, 정직함 +1.

권장: 2-2와 2-3을 먼저 고쳐 폰트 교체를 안전하게 만든 뒤, 셋째를 즉시 하고, 둘째를 다음에 한다.

## 3. 다음으로

### 3-1. 죽은 서버 라우트 하나가 남아 있다 — 확인됨

`/api/generate-html`은 UI 어디에서도 호출하지 않는다. `src/lib/html-generator.ts`(501줄) 전체가 이 라우트를 통해서만 도달 가능하다.

두 갈래다. **살리든가, 지우든가.** 살릴 가치는 있다 — 브라우저에서 바로 띄우는 발표 모드는 PowerPoint가 없는 교실에서 유용하고, 클릭 애니메이션도 이미 구현돼 있다. 지운다면 서버 라우트가 하나(`generate-pptx`)만 남고, 그마저 클라이언트로 옮기면(pptxgenjs는 브라우저에서 돈다) **완전 정적 사이트**가 되어 Vercel 함수 제한과 무관해진다.

지금처럼 방치하는 것만 답이 아니다. 도달 불가능한 501줄은 유지보수 부채이고, 아래 3-2의 취약점이 잠복하는 곳이기도 하다.

### 3-2. 프로젝트 JSON을 통한 HTML 속성 주입 — 조건부

`src/lib/html-generator.ts:113, 123, 135`

```ts
`<span class="ann-wrap step" ... data-marker="${ann.markerType}" style="--ann-color:${ann.color}">`
```

`ann.color`와 `ann.markerType`은 이스케이프를 거치지 않는다(`escapeHtml`은 `ann.content`에만 적용된다). 그리고 `readJsonFile`(`useEditorState.ts`)은 `data.genre`와 `data.slides` 존재 여부만 검사한다. 즉 악의적인 `.json` 프로젝트 파일을 받아 불러오면 `color: 'red" onmouseover="…'` 같은 값이 속성을 탈출한다.

**현재는 실제로 악용 불가능하다** — 3-1에서 봤듯 HTML 생성 경로가 UI에서 끊겨 있기 때문이다. 하지만 HTML 발표 모드를 살리는 순간 살아난다. 그리고 생성된 HTML은 교사가 동료에게 전달하도록 만든 파일이다.

고칠 방향: 불러오기 시점에 검증한다(`markerType`은 화이트리스트, `color`는 `/^#[0-9a-f]{6}$/i`). 속성 보간에도 이스케이프를 적용한다. 난이도 S. 3-1에서 "살린다"를 택하면 **함께 해야 한다.**

### 3-3. 도메인 핵심에 테스트가 하나도 없다

`slide-splitter`, `annotation-matcher`, `pptx-generator`, `pptx-geometry`, `pptx-animation`, `font-metrics`, `html-generator` — 전부 0건. 현재 15개 테스트는 전부 BYOK 관련 모듈에 있다. 위 1-1과 1-2를 프로브 두 개로 5분 만에 잡아냈다는 사실이 이 항목의 근거다.

첫 번째 스위트로 무엇을 쓸지 구체적으로:

`annotation-matcher.test.ts`
- 중복 `targetText` 두 개가 서로 다른 출현에 매칭된다 (지금은 실패 → 1-1의 회귀 테스트)
- 슬라이드 경계를 걸치는 주석이 양쪽에 나타난다 (지금은 실패 → 1-2의 회귀 테스트)
- 곡선 따옴표(`‘’`)만 다른 `targetText`가 정규화 경로로 매칭된다
- 원문에 없는 `targetText`는 `unmatched`로 떨어진다

`slide-splitter.test.ts` — 골든 케이스 6개

### 3-4. 옛한글 폰트 3MB가 매 페이지 로드에 실린다 — 확인됨

`src/app/layout.tsx:17-21`에서 `NanumBarunGothic-YetHangul.otf`(3,056,264 바이트)를 `next/font/local`로 불러온다. `display: "swap"`은 있으나 `preload`는 기본 `true`다. 교실 와이파이에서 첫 방문 3MB는 뼈아프다. 게다가 옛한글은 극소수 작품에서만 필요하다.

고칠 방향: `preload: false`로 두고, 옛한글이 실제로 필요한 화면(본문 입력칸·슬라이드 표시 영역)에서만 쓰이도록 유지한다. 더 나아가면 `pyftsubset`으로 필요한 글리프만 남긴 서브셋 + woff2로 바꾸면 수백 KB로 줄어든다. 저장소에 있는 6MB `.ttf`는 어디서도 쓰이지 않으니 지운다.

### 3-6. `useHistory`가 렌더 중 ref를 읽는다

`src/hooks/useHistory.ts:66-67`

```ts

이것이 lint 오류 2건의 정체다. 지금은 `bump()`가 강제로 리렌더를 일으켜 우연히 동작한다. React의 동시성 렌더링에서는 실행 취소 버튼의 활성/비활성 상태가 실제 스택과 어긋날 수 있다. 또 `push`가 `current`를 의존성으로 잡으므로(:35) 한 틱에 `push`를 두 번 부르면 중간 상태 하나가 히스토리에서 누락된다.

## 4. 제품 관점에서 정직하게

**추출 결과가 틀렸을 때를 위한 장치가 없다.** 어디에도 "AI가 추출한 내용이니 확인하세요"라는 문구가 없고, 교사가 자기가 쓴 주석과 Gemini가 뽑은 주석을 화면에서 구분할 방법이 없다. 잘못 뽑힌 주석이 교실 스크린에 뜨는 순간이 이 도구의 첫 번째 신뢰 위기다. `matchAnnotationsToText`가 unmatched를 경고로 보여주긴 하지만, 그건 "못 찾은 것"이지 "틀리게 찾은 것"이 아니다. 1-1의 버그는 정확히 후자를 만들어낸다 — 조용히, 그럴듯하게.

제안: 추출된 주석에 출처 배지를 달고(AI/직접), 생성 직전 "AI 추출 주석 N개를 확인하셨습니까" 체크를 한 번 넣는다. 그리고 슬라이드 미리보기를 붙인다. 지금은 `.pptx`를 열어봐야 결과를 안다.

**같은 PDF가 매번 같은 결과를 주지 않는다.** `temperature: 0.1`이고, 2라운드 재검증은 unmatched가 있을 때만 돈다(`gemini-client.ts`). 따라서 동일 입력에 대해 라운드 수와 결과가 달라질 수 있다. 교사가 "어제 나온 그 슬라이드"를 재현하려 하면 실패한다. 프로젝트 JSON 저장이 사실상 유일한 재현 수단인데, 그 저장이 1-3에서 보듯 설정을 날린다.

**핵심 가치 제안을 스스로 의심해볼 필요가 있다.** 이 도구의 가장 어려운 부분(PDF에서 주석 추출)이 가장 신뢰하기 어려운 부분이다. 반면 가장 확실하게 가치 있는 부분은 **클릭 애니메이션이 붙은 주석 슬라이드를 자동 조판하는 것**이다 — 이건 교사가 손으로 하면 정말 오래 걸리고, 결과도 결정적이다. Mode B(직접 입력)가 사실 이 도구의 심장일 수 있다. PDF 추출은 편의 기능이고. 만약 그렇다면 온보딩의 우선순위가 바뀐다: API 키 없이 쓸 수 있는 Mode B를 첫 화면에 놓고, 키가 필요한 PDF 추출을 "고급"으로 내리는 것이다. 지금은 Mode C가 "(추천)"으로 기본 선택돼 있어, 첫 방문자가 곧바로 API 키 벽에 부딪힌다.

이 판단은 실제 교사 몇 명에게 두 모드를 써보게 하면 하루 만에 결론이 난다.

- `인디스쿨`은 초등교사 전용 폐쇄 커뮤니티이므로 고등 국어 도구의 확산 경로가 될 수 없다.
- 산출물이 정의상 교사용 교과서의 주석을 담으므로, 그것을 커뮤니티에 공유하는 행위가 저작권법이 가장 명확히 금지하는 지점이다.

## 6. 디자인 대안

### 6-1. 진단: 지금 색은 고른 색이 아니다

현재 UI는 `#6B3F26` 갈색 글씨에 `#EEDDD0` 크림 배경이다. Claude Code의 `frontend-design` 지침은 AI가 어떤 브리프에나 습관적으로 꺼내는 세 가지 기본값을 명시하는데, 그 **1번이 "따뜻한 크림색 배경 + 고대비 세리프 + 테라코타 액센트"**다. 이 앱의 팔레트가 정확히 거기에 있다.

국어 문학 수업을 위해 선택된 색이 아니라는 뜻이다.

더 결정적인 건 따로 있다. 이 도구가 **만들어내는 산출물**의 색은 이미 정해져 있다 — `pptx-constants.ts:55`의 마커 색 `#294C67`(교사용 교과서의 파란 밑줄), `html-generator.ts:15`의 요약 상자 배경 `#E8EFF5`, 슬라이드 배경 흰색. 그런데 **도구의 껍데기는 그 산출물과 아무 상관 없는 갈색이다.** 교사는 갈색 화면에서 작업해 파란 슬라이드를 뽑는다. 도구가 자기가 만드는 물건과 다른 언어로 말하고 있다.

### 6-2. 방향 A — 교정지 (권장)

> 동작하는 목업: [`docs/mockups/direction-a.html`](mockups/direction-a.html) — 브라우저로 열면 된다. 주석이나 여백 노트에 마우스를 올리면 지시선이 파랗게 살아나고, **리허설** 버튼(또는 <kbd>R</kbd>)을 누르면 교실에서 벌어질 클릭 시퀀스가 재생된다. <kbd>Esc</kbd>로 빠져나온다.

**Thesis.** 이 도구는 교사용 교과서의 여백을 스크린으로 옮긴다. 그러므로 도구는 그 교과서의 잉크와 종이로 만들어져야 한다.

교사용 교과서는 실재하는 물건이고 자기 시각 언어를 갖고 있다. 파란 밑줄, 빨간 동그라미, 「꺾쇠」, ▶ 여백 요약, 가는 괘선. 우리는 그것을 새로 발명할 필요가 없다. 이미 `MARKER_COLOR`로 코드 안에 들어와 있다.

**Type.**
- 본문(작품 텍스트): 세리프. 읽는 대상이 문학이기 때문이지 분위기 때문이 아니다. **다만 이 창은 옛한글을 그려야 하므로** 현재의 `NanumBarunGothic-YetHangul`을 그대로 쓰되 `.kor-text`에만 스코프하고 `preload: false`로 돌린다(3-4 해결).
- 크롬(버튼·라벨·패널): **Pretendard** — 조용한 산세리프. 껍데기는 말을 걸지 않는다.
- 유틸리티(글자 수, 슬라이드 번호, 단축키): **IBM Plex Mono** 또는 Pretendard의 tabular-nums. 숫자가 흔들리지 않아야 한다.

세리프 + near-white는 기본값 1번과 스칠 수 있다. 벗어나는 장치는 셋이다. 종이가 크림이 아니고, 액센트가 파랑이며, **세리프는 작품 텍스트 창에만 갇혀 있다.** 나머지 화면은 전부 산세리프다.

**Signature — 여백 주석 (marginalia).**

이 앱이 기억될 단 하나의 요소. 주석은 오른쪽 패널의 목록이 아니라, 자기가 가리키는 행 **옆 여백에** 매달린다. 가는 지시선(leader)이 주석과 본문의 정확한 구간을 잇는다. 교사용 교과서가 실제로 생긴 모양 그대로다.

지금은 주석↔본문 관계가 목록의 순번을 눈으로 대조해야 하는 조회 작업이다. 여백에 걸면 그 관계가 **공간이 된다.** 이것이 이 도구가 만드는 산출물의 형태이기도 하다.

```

**Motion — 리허설.** 유일한 연출된 순간. 단축키 하나로 에디터가 교실에서 벌어질 클릭 시퀀스를 그대로 재생한다. 마커가 뜨고, 주석이 뜨고, 다음 마커가 뜬다. `.pptx`와 같은 순서로.

이건 장식이 아니다. 지금 에디터에는 **미리보기가 없어서** 교사가 PowerPoint를 열기 전까지 결과를 모른다. 리허설은 그 구멍을 메우면서, 동시에 이 제품의 교육적 핵심 — 한 번에 하나씩 드러내기 — 을 도구 안에서 체험하게 만든다. 하나의 장치가 두 일을 한다.

`prefers-reduced-motion`에서는 전체 시퀀스를 정지 상태로 한 번에 보여준다.

**Copy.** 사용자가 조작하는 대상의 이름으로 부르고, 능동태로 쓰고, 동작 이름을 흐름 내내 유지한다.

| 지금 | 바꿀 것 | 이유 |
|------|---------|------|
| 주석 추출 + 분할 | PDF에서 주석 가져오기 | 두 동작을 한 버튼이 하고 있었다 |
| 슬라이드 분할 | 슬라이드로 나누기 | 명사가 아니라 동사 |
| PPT 생성 | PPT 내보내기 | 만든 게 아니라 꺼내는 것 |
| PDF 주석 추출에는 본인의 Gemini API 키(무료 발급)가 필요합니다. | PDF에서 주석을 가져오려면 본인의 Gemini 키가 필요합니다. 발급은 무료이고 1분 걸립니다. | 비용과 시간을 먼저 말한다 |
| API 키가 유효하지 않습니다. | 이 키로는 요청할 수 없습니다. 키를 다시 확인해 주세요. | 오류는 사과하지 않고, 다음 행동을 말한다 |
| (키 없는 빈 화면) | 직접 입력으로 지금 시작할 수 있습니다. 키는 PDF를 쓸 때만 필요합니다. | 빈 화면은 초대장이다 |

마지막 줄이 가장 중요하다. Mode B는 키 없이 완전히 동작하는데 지금 첫 화면은 그 사실을 말하지 않는다.

**감수한 위험.** 여백 레이아웃은 본문 폭을 좁히고, 지시선 계산이 필요하며, 좁은 화면에서 무너진다. 패널 목록보다 확실히 어렵다. 그럼에도 택하는 이유는 그것이 이 소재가 실제로 생긴 모양이고, 주석과 본문의 관계를 조회가 아니라 시선 이동으로 바꾸기 때문이다. 태블릿 이하에서는 여백을 접어 본문 아래 카드로 떨어뜨린다.

### 6-3. 방향 B — 판서 (기각)

칠판 초록(`#16211D`) 위 분필 흰색, 분필 노랑 액센트. 에디터가 곧 투사면이 되어 교실에서 보일 그대로를 편집한다.

**기각한 이유는 셋이다.** 첫째, 어두운 캔버스 + 밝은 단일 액센트는 `frontend-design`이 지목한 **기본값 2번**이다. 기본값 1번을 피하려다 2번으로 걸어 들어가는 셈이다. 둘째, 교사는 밝은 교무실에서 긴 텍스트를 편집한다. 어두운 배경은 그 작업에 적대적이다. 셋째이자 결정적으로, **산출물 슬라이드의 배경은 흰색이다**(`pptx-constants.ts:28`, `SLIDE_BG_COLOR = "FFFFFF"`). 어두운 에디터는 결과물에 대해 거짓말을 한다.

방향 A가 가져갈 것: **투사면을 편집한다**는 발상. 리허설 모드가 그 발상의 실현이다.

### 6-4. 남은 클리셰

방향 A도 완전히 결백하진 않다. 여백의 가는 지시선과 괘선은 **기본값 3번(브로드시트: 헤어라인 괘선, 라운드 0, 조밀한 단)** 과 이웃한다. 경계는 이렇게 긋는다 — 지시선은 장식이 아니라 주석과 구간을 잇는 기능선이고, 카드에는 충분한 라운드를 주며, 본문은 한 단으로 넉넉하게 흐른다. 신문이 아니라 교과서다.

숫자 마커(①②③)는 쓴다. 다만 그것이 정당한 이유는 **클릭 순서가 실제로 순서이기 때문**이다. 등장 차례가 정보다. 순서가 아닌 것에 번호를 붙이지는 않는다.

### 6-5. 이 디자인이 고치는 것

우연이 아니라 설계다. 방향 A는 3-5의 접근성 실패를 팔레트 차원에서 제거하고("키 설정" 버튼이 `#294C67` 위 흰 글씨가 되면 2.15:1 → 9.04:1), 4장의 "AI 출처 구분 없음" 문제에 자리를 만들며(여백 주석에 출처 배지를 다는 것은 자연스럽다), 3장의 "미리보기 없음"을 리허설로 해결한다.

색을 바꾸는 일이 아니다. 도구가 자기가 만드는 물건과 같은 언어를 쓰게 만드는 일이다.

## 7. 권장 순서

1. **`SavedProject`에 `pptSettings` 추가** (1-3) — 반나절, 무손실 저장. 지금 사용자 작업물이 조용히 훼손되고 있다.
2. **"키 설정" 버튼 대비 수정**(3-5) — 한 줄. 키 없는 첫 방문자 전원에게 영향한다.
3. **슬라이드 병합 시 `order` 재부여**(1-5) — 한 줄. 클릭 순서가 뒤엉키는 것을 막는다.
4. **매처 회귀 테스트 2개를 먼저 쓰고**(3-3), 빨간 상태에서 **중복 `targetText` 매칭을 고친다**(1-1) — 이 도구에서 가장 조용하고 가장 위험한 버그.
5. **경계 걸침 주석 처리**(1-2) — 4번의 테스트 인프라를 그대로 쓴다.
6. **폰트 지뢰 제거**(2-2, 2-3) — 죽은 상수를 지우고 `lineStepRatio`를 메트릭스에서 읽게 한다. 두 줄. 이후 폰트 교체가 안전해진다.
7. **`/api/generate-html`을 결정한다**(3-1) — 살리면 3-2를 함께, 지우면 완전 정적화의 길이 열린다.

디자인(6장)은 큰 덩어리다. 다만 2번은 그 팔레트의 첫 조각이므로 지금 해도 손해가 없다. 여백 주석과 리허설은 별도 계획으로 다룰 일이다.

옛한글 폭 계산(1-4), 폰트 서브셋(3-4), `useHistory` 정리(3-6)는 언제 해도 좋다.
